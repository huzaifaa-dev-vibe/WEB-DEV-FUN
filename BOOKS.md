# BOOKS

> Books worth reading, with a one-line "why". Sorted by topic, then by recommended order within topic.

Books marked **[free]** are available free online. Books marked **[reference]** are skim-on-demand, not cover-to-cover.

---

## Career & Engineering Culture

| Book | Why |
|---|---|
| *The Pragmatic Programmer* — Hunt & Thomas | The book that turns coders into engineers. Read every 5 years. |
| *Software Engineer's Guidebook* — Gergely Orosz | Modern, practical, covers L3-L6. |
| *Staff Engineer* — Will Larson | What "staff" actually means and how to operate at that level. |
| *The Software Craftsman* — Sandro Mancuso | Professionalism in code. |
| *The Clean Coder* — Robert C. Martin | Ethics, saying no, professionalism. |
| *Debugging* — David J. Agans | 9 rules for finding bugs. Short, punchy. |
| *How to Be a Programmer* — Robert L. Read [free] | https://braydie.com/how-to-be-a-programmer/ |
| *The Passionate Programmer* — Chad Fowler | Career strategy. |

## Software Design & Architecture

| Book | Why |
|---|---|
| *Designing Data-Intensive Applications* — Martin Kleppmann | The bible of distributed systems. Read before designing anything at scale. |
| *A Philosophy of Software Design* — John Ousterhout | Deep modules, narrow interfaces. Short and excellent. |
| *Refactoring* — Martin Fowler | Catalog of smells + transformations. [reference] |
| *Clean Code* — Robert C. Martin | Read critically — some advice aged poorly. The naming/structure chapters are gold. |
| *Code Complete* — Steve McConnell | The encyclopedic reference. [reference] |
| *Domain-Driven Design* — Eric Evans | Heavy. Read Part I; reference the rest. |
| *Implementing DDD* — Vaughn Vernon | More accessible than Evans. |
| *Patterns of Enterprise Application Architecture* — Fowler [reference] | Catalog of architecture patterns. |
| *Head First Design Patterns* | Friendly intro to GoF. |
| *Design Patterns* — GoF | The classic. Dense. [reference] |
| *97 Things Every Software Architect Should Know* | Bite-sized wisdom. |
| *Building Evolutionary Architectures* — Ford, Parsons, Kua | Architecting for constant change. |
| *Software Architecture: The Hard Parts* — Ford, Richards, Sadalage, Dehghani | Trade-offs at scale. |

## Bonus: Reading Order Suggestions

### For a junior dev (year 1)
1. *Eloquent JavaScript* (free)
2. *The Pragmatic Programmer*
3. *Refactoring UI*
4. *Don't Make Me Think*

### For a mid-level engineer (year 2–4)
1. *Designing Data-Intensive Applications*
2. *A Philosophy of Software Design*
3. *The Manager's Path* (even if you stay IC)
4. *Site Reliability Engineering* (free)

### For a senior / staff engineer
1. *Staff Engineer*
2. *Designing Distributed Systems*
3. *Accelerate*
4. *Observability Engineering*

### For an AI engineer
1. *Build a Large Language Model (From Scratch)*
2. *Designing Machine Learning Systems*
3. *AI Engineering*
4. *Prompt Engineering Guide* (free)
